# BACKLOG 8.23 â€” Cross-track curve selection (Select tool)

## Context

Today only the active track's curves are clickable. To select a curve on another track the user has to first switch tracks via the Tracks panel, then click â€” clunky when several tracks overlap in the same pitch range and the user just wants to grab whichever line their eye is on. The renderer already draws every non-muted track's curves, so the visual is in place; only the Select-tool hit-test is single-track-scoped.

Make the Select tool's plain click hit any visible (non-muted) curve on any track. The clicked curve becomes the only selection AND its track becomes the active track. Shift-click stays locked to the active track so multi-select can't be accidentally torpedoed by a stray click on another track.

## Design

### Hit-test (interaction.ts)

In `handleSelectClick` ([src/canvas/interaction.ts:630](src/canvas/interaction.ts)):

- **Plain click (no shift):** widen hit-testing to iterate every non-muted track in `comp.tracks` (same iteration order as the renderer in [src/main.ts:2735](src/main.ts) so visual z-order matches click priority). Run the existing three-phase test (anchor points, then bezier handles, then segments) across that wider set. When a hit is found:
  - If the hit's curve is on the active track: existing behavior unchanged.
  - If on a different track: call `store.setSelectedTrack(hitTrackId)` first (which clears selection + retargets the primary planchette), THEN `store.setSelectedCurve(hitCurveId)` and (if it was an anchor-point hit) `store.setSelectedPoint(idx)`. The track-first ordering matters because `setSelectedTrack` clears `selectedCurveIds`/`selectedPointIndex`.
  - If no curve is hit: existing deselect path (clear selection, don't change track).

- **Shift click:** unchanged. Only the active track is hit-tested. A shift-click on empty active-track space (which now means "any space not occupied by an active-track curve" â€” even if a non-active-track curve is under the cursor) is a no-op for selection. This is the explicit user-confirmed behavior: shift never changes the active track and never reaches across.

- **Bezier handle hit-test (lines 665â€“698):** stays single-curve / single-track. Handles only render for the active selection, so they only exist on the active track anyway.

The cross-track scan lands the hit-curve's parent track id alongside the curve in the existing return shape (or a small refactor into a `{ track, curve, kind, idx? }` discriminated result so `handleSelectClick` can branch on it without re-finding the track). I'll thread the parent track through whichever helper finds the hit so we don't have to do a second `comp.tracks.find(t => t.curves.includes(curve))` lookup later.

### Group expansion

`rebuildTransformBox` ([src/canvas/interaction.ts:730](src/canvas/interaction.ts)) takes the active track and runs `expandSelectionToGroups` on it. After we switch tracks for a cross-track click, the active track is the *new* one, so `rebuildTransformBox` naturally operates on the new track's curves and expands any chord group there. No changes needed.

### Visual dim (renderer)

In [src/canvas/curve-renderer.ts](src/canvas/curve-renderer.ts), extend `renderCurves` with a sixth parameter `isActiveTrack: boolean`. When false, wrap the per-curve drawing in `ctx.globalAlpha = 0.45` (then restore). This dims the line + control points + handles together. Keep stroke widths and colors unchanged â€” the alpha alone is enough to push non-active tracks visually back without changing their identity.

In [src/main.ts:2735â€“2748](src/main.ts), the loop already computes `isActiveTrack`; just thread it into the `renderCurves` call.

Possible follow-up if 0.45 looks wrong in practice: tune to 0.35â€“0.55 during verification.

### Tools that stay single-track

Per the user's answer: only the Select tool changes. Delete, Scissors, and Draw all keep `getSelectedTrack()` scoping. They already respect the active track and that's the right ergonomics for editing operations (no surprise cross-track deletes).

## Edge cases

- **Click lands on overlapping curves from two tracks at once:** hit-test iterates tracks in render order (forward through `comp.tracks`); within a track, the existing first-match precedence stands. So the visually topmost curve wins, which matches user expectation.
- **Click on a muted-track curve:** the renderer doesn't draw muted tracks, and the hit-test loop filters `track.muted`, so muted curves are unclickable. This matches the user's "ignore muted ones" requirement.
- **Cross-track click while a transform-box drag is in progress:** can't happen â€” drag captures the mouse and `handleSelectClick` only fires on click-up of a non-drag click. Existing drag-vs-click discrimination stays.
- **Cross-track click on a chord-group sibling:** track switches, sibling becomes selected, then `rebuildTransformBox` on the new track expands to the whole group there. Group stays coherent.
- **Shift-click on a non-active-track curve:** falls through to the empty-space branch (no active-track hit). Selection is unchanged. The user keeps their multi-select intact, as requested.
- **Click on the active track's empty space when a non-active curve is visually under the cursor:** plain click selects that non-active curve (cross-track hit). Shift-click does nothing (active-track-only scope misses it). Both are coherent with the rules above.
