# BACKLOG 8.13 — Per-voice octave offsets (Harmonic Prism Voicing)

## Context

The Harmonic Prism today exposes only global chord-spec controls — Stacking, Quality, Voices, Tuning, Direction, Octaves. There's no way to lift one voice an octave to make a spread voicing or drop a bass voice for a wide spacing. Per-voice octave offsets fill that gap, and as a side effect cover inversions: a 1st-inversion triad is just `[+1, 0, 0]`; 2nd inversion is `[+1, +1, 0]`.

Per-voice offsets land at the chord-spec level. They affect Harmonic Prism live perform (held voices retune) and future placements; existing placed clusters keep their baked Y values, matching how every other chord-spec change already works.

## Design

### Data model

Extend `ChordSpec` in [src/utils/harmonics.ts](src/utils/harmonics.ts):

```ts
export interface ChordSpec {
  stacking: StackingStyle;
  quality: ChordQuality;
  numVoices: NumVoices;
  tuning: TuningSystem;
  direction: Direction;
  /** ±octave offset per voice index (0 = root). Length up to numVoices.
   *  Missing or extra entries are treated as 0. New in 8.13. */
  voiceOctaveOffsets: number[];
}
```

`DEFAULT_CHORD_SPEC` gets `voiceOctaveOffsets: []`. Empty array is the "all zeros" canonical form so an unset chord spec round-trips identically.

### Pitch computation

`chordOffsets(spec)` (currently lines 206–210) gains one extra step at the end:

```ts
export function chordOffsets(spec: ChordSpec): number[] {
  const base = spec.tuning === '12-TET' ? chordOffsets12TET(spec) : chordOffsetsJI(spec);
  const directed = applyDirection(base, spec.direction);
  const oct = spec.voiceOctaveOffsets;
  return directed.map((s, i) => s + ((oct[i] ?? 0) * 12));
}
```

Octave offsets apply *after* direction so the user's mental model stays consistent: "voice 2 up an octave" always means +12 semitones from where that voice would otherwise land, regardless of whether the chord is going up, down, or symmetric.

This single change naturally flows through every existing chord consumer:
- Projection echoes ([src/canvas/projection-renderer.ts:53](src/canvas/projection-renderer.ts)) re-render at the new offsets.
- Draw-mode placement ([src/canvas/interaction.ts:923](src/canvas/interaction.ts)) places clusters with the new Y values.
- Live perform retune (`updateHarmonyVoices` in [src/main.ts:2000](src/main.ts)) shifts held voices instantly when the user changes an offset mid-perform.

No call-site changes; just the math step inside `chordOffsets`.

### UI

In [src/ui/prism-panel.ts](src/ui/prism-panel.ts), append a new VOICING subsection at the bottom of the panel (after the Octaves row). Use the same `panel-header` style as other subsections so it visually groups with TOOLS / SNAP etc. above.

Inside the subsection, render `numVoices` rows — one per voice — re-rendered whenever the spec changes:

```
VOICING
Voice 1 (root)    [number input, ±2, default 0]
Voice 2           [number input, ±2, default 0]
Voice 3           [number input, ±2, default 0]
...
```

Number inputs use `min="-2" max="2" step="1"` to stay readable; the user can stack two octaves either direction without the UI bloating. Pattern mirrors the existing Octaves ± input ([src/ui/prism-panel.ts:89](src/ui/prism-panel.ts)). The voice 0 row is labeled `Voice 1 (root)` so it's clear that bumping it up is a valid move (raises the entire chord plus its harmonics together — useful for inversions).

The voicing subsection re-renders on every store notify (the panel already subscribes for the spec dropdowns), so changing `numVoices` automatically grows or shrinks the row count. When the user picks a smaller `numVoices`, the underlying `voiceOctaveOffsets` array isn't truncated — extra entries just stay there silently and become live again if the user bumps `numVoices` back up. This matches how the spec already silently retains other fields.

### Change handlers

Each row's `input` event reads the current `voiceOctaveOffsets`, splices in the new value at that voice index, and calls `store.setPrismChordSpec({ voiceOctaveOffsets: [...] })`. Empty trailing zeros are preserved (don't trim) so the array length always equals at least the highest-set voice index + 1; this keeps round-trip JSON deterministic.

### Persistence

`loadChordSpecPref` in [src/state/store.ts:71](src/state/store.ts) gets one more per-field fallback:

```ts
voiceOctaveOffsets: Array.isArray(parsed.voiceOctaveOffsets)
  ? parsed.voiceOctaveOffsets.filter((n: unknown): n is number => typeof n === 'number' && Number.isFinite(n))
  : defaultSpec.voiceOctaveOffsets,
```

Old localStorage entries without the field open with `[]` (all-zeros), so the change is fully backwards-compatible. No version bump.

`saveChordSpecPref` is unchanged — it `JSON.stringify`s the whole spec.

The composition file does not store the chord spec (only localStorage does), so there's no composition schema impact.

## Verification

Dev server at http://localhost:5187.

Direct math checks via `preview_eval` (mirroring how 8.19 / 8.2 / 8.23 were verified):
- `chordOffsets({ ...default, voiceOctaveOffsets: [] })` → `[0, 4, 7]` (unchanged for major triad).
- `chordOffsets({ ...default, voiceOctaveOffsets: [0, 1, 0] })` → `[0, 16, 7]` (voice 2 up an octave — 1st inversion shape).
- `chordOffsets({ ...default, voiceOctaveOffsets: [0, 0, -1] })` → `[0, 4, -5]` (voice 3 dropped an octave — open spacing).
- `chordOffsets({ ...default, direction: 'down', voiceOctaveOffsets: [0, 1, 0] })` → voice 2 still gets +12 from where the descending chord placed it.
- Spec with a 4th-voice offset but `numVoices: 3` → only 3 offsets returned; the 4th-voice entry doesn't leak.

Manual / preview-tool checks:
- VOICING subsection appears at the bottom of the Harmonic Prism panel with rows matching `numVoices`. Default values all 0.
- Bump Voice 2 to +1 → projection echoes (Ctrl+H on a curve) shift voice-2 echoes up an octave; voice-1 and voice-3 echoes unchanged.
- In Draw mode with chord placement on, click to place a new cluster → voice 2's curve sits an octave above where it would have without the offset.
- Live perform: hold LMB to start the chord, change Voice 2 from 0 to +1 mid-hold → the held voice retunes up an octave instantly (audible + planchette moves).
- Existing placed clusters: confirm their Y values do NOT move when offsets change (chord spec is "going forward" only, matches Stacking/Quality/Tuning behavior).
- Change `numVoices` 3 → 5 → the subsection grows two new rows defaulting to 0; previously-set offsets for voices 1–3 persist.
- Reload the page → offsets persist via localStorage.
- Old (pre-8.13) localStorage entry — synthesize one without the field — loads as all-zeros, no console errors.
- TypeScript clean (`npx tsc --noEmit`); no console / server errors.
