# 8.20 — Record AFK timer should respect loop / future content

## Context

The perform-engine AFK timeout (`afkTimeoutMs = 20_000` ms — see [src/main.ts:1894](src/main.ts) and the gate in [src/canvas/performance-engine.ts:145-147](src/canvas/performance-engine.ts)) auto-stops the recording session whenever record is armed and no input activity is registered for 20 seconds. Today it fires even when the session has a meaningful reason to keep waiting:

- **Loop is on** — the user is intentionally recording over loops; auto-stop ruins that.
- **Playhead hasn't reached the rightmost control point yet** — the playhead is still scrolling through existing content; the user is naturally waiting for "blank space" to perform into.
- **MIDI input is the active input source** — currently only LMB-down and `captureSample` mark activity ([performance-engine.ts:70-72,100-107](src/canvas/performance-engine.ts)). MIDI key presses don't, so an armed MIDI-only session AFKs out even while the user is actively playing keys.

All three cases should refresh the AFK timer. Once the suppressing condition lifts (loop disabled, playhead crosses the rightmost point, MIDI keys go quiet), the user should get a full `afkTimeoutMs` window from that moment before AFK fires — not get immediately disarmed because the AFK clock had been accumulating in the background.

## Approach

Refresh `lastActivityAt` from the **consumer side** using the engine's already-exported `markActivity(now)` method ([performance-engine.ts:29,83-85](src/canvas/performance-engine.ts)). Two consumer touch points in [src/main.ts](src/main.ts):

1. **`tickComposePerform`** — every tick, mark activity if Loop is on or the playhead is still before the rightmost control point.
2. **MIDI input callbacks** — mirror the LMB-down pattern: mark activity on every `noteOn` and `noteOff`.

This keeps domain knowledge ("loop", "composition rightmost", "MIDI input") out of the perform engine and avoids expanding `TickArgs`. Reuses [`getCompositionLength()`](src/model/composition.ts:46) (already used in adjacent code paths in main.ts at lines 2340 and 2399, so the per-frame O(N points) cost is already paid in this area).

## Implementation

### File 1: [src/main.ts](src/main.ts) — two edits

**Edit A — `tickComposePerform()` ([src/main.ts:2312-2335](src/main.ts))**: insert a suppression block before `composeEngine.tick({...})`:

```ts
function tickComposePerform() {
  const st = store.getState();
  const g = st.performance;
  const anyArmed = g.recordArmed || st.midiArmedTrackId !== null;
  const playbackBeat = playback.getPositionBeats();

  // Keep AFK timer fresh while there's a meaningful reason to keep waiting:
  // (a) Loop is on (intentional record-over-loops), or (b) the playhead hasn't
  // crossed the rightmost control point yet (still future content to record over).
  if (anyArmed && g.phase === 'playing' && playback.isPlaying()) {
    const rightmost = getCompositionLength(st.composition);
    if (playback.isLoopEnabled() || playbackBeat < rightmost) {
      composeEngine.markActivity(performance.now());
    }
  }

  composeEngine.tick({
    now: performance.now(),
    audioNow: getAudioContext().currentTime,
    isPlaying: playback.isPlaying(),
    phase: g.phase,
    recordArmed: anyArmed,
    countdownStartedAt: g.countdownStartedAt,
    playbackBeat,
    onCountdownElapsed: startComposePerformPlayback,
    onLoopWrap: () => {
      if (g.recordArmed && composeEngine.isLmbDown()) finalizeComposeRecordedCurves();
      finalizeAllInFlightMidiVoices();
    },
    onAfkTimeout: composePerformStop,
  });
}
```

**Edit B — MIDI callbacks ([src/main.ts:702-744](src/main.ts))**: mirror the LMB-down pattern by marking activity in both noteOn and noteOff. Insert one line after `ensureResumed()` in `onNoteOn` (around line 713) and one line at the top of `onNoteOff` (around line 740):

```ts
midiInput.onNoteOn((note, velocity) => {
  // ... existing guards / track + tone lookup
  ensureResumed();
  composeEngine.markActivity(performance.now());  // ← add
  // ... existing preview + recording logic
});

midiInput.onNoteOff((note) => {
  composeEngine.markActivity(performance.now());  // ← add
  preview.stopDrawPreview(`midi-${note}`);
  finalizeMidiVoice(note);
});
```

Notes:
- The `tickComposePerform` gate uses `anyArmed && phase === 'playing' && isPlaying()` to mirror the engine's own AFK gate ([performance-engine.ts:137,145](src/canvas/performance-engine.ts)) — no point marking activity when AFK can't fire anyway.
- `playbackBeat < rightmost` evaluates `false` for empty compositions (`rightmost === 0`), so empty canvases keep their existing AFK behavior.
- At the rightmost-point boundary itself (`playbackBeat === rightmost`), suppression lifts. Acceptable — there's no future content left at that exact point.
- The MIDI callbacks mark activity unconditionally (matching `onLmbDown` in [performance-engine.ts:70-72](src/canvas/performance-engine.ts)). `markActivity` is harmless when not armed since the AFK gate is `recordArmed`-conditional.
- For *sustained* MIDI notes (held > 20s), the existing per-frame `captureSample` calls during recording already mark activity ([performance-engine.ts:106](src/canvas/performance-engine.ts)) via the MIDI sample-capture path from Phase 8.11. So the noteOn/noteOff marks are sufficient — no per-frame planchette polling needed.

No changes needed in [src/canvas/performance-engine.ts](src/canvas/performance-engine.ts), no `TickArgs` expansion, no new imports (`getCompositionLength` is already imported in main.ts).

## Verification

Run dev server (`npm run dev` → port 5187). Test cases:

1. **Loop on, AFK** — Enable Loop, arm record, sit idle > 20s. Recording must NOT auto-stop.
2. **Loop off, playhead before rightmost, AFK** — Draw a curve out to ~beat 50, place playhead at beat 0, arm record, sit idle > 20s. Must NOT auto-stop while playhead < rightmost.
3. **Loop off, playhead past rightmost, AFK** — Same setup but playhead past the last control point, arm record, sit idle > 20s. SHOULD auto-stop (regression check on existing behavior).
4. **Crossing the boundary mid-AFK** — Loop off, playhead before rightmost, arm record, sit idle through the crossing. After playhead crosses rightmost, AFK should fire ~20s later — NOT immediately. (Confirms the "fresh window after lift" behavior.)
5. **MIDI-only armed, playing keys** — MIDI-arm a track, no LMB, just press MIDI keys every 10–15s for over a minute. Must NOT auto-stop.
6. **MIDI-only armed, idle keys past rightmost** — MIDI-arm, playhead past rightmost, loop off, no MIDI input for > 20s. SHOULD auto-stop.
7. **Sustained MIDI note past rightmost** — MIDI-arm, playhead past rightmost, loop off, hold a single MIDI note for > 20s. Must NOT auto-stop (relies on `captureSample` activity marks during MIDI recording).
8. **help.html** — Open help, navigate to Perform & Record → Recording, confirm the new wording reads correctly.

Sanity check: normal LMB-held performing should continue working unchanged in all cases (the gate only adds activity marks; doesn't remove any existing behavior).
