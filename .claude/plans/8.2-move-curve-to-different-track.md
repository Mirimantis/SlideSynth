# BACKLOG 8.2 — Move curve to a different track

## Context

Currently a curve lives on whichever track it was drawn on. There's no way to relocate it short of copy-paste (which loses identity, regenerates ids, and requires switching the active track first). We want a one-click move from the Object Properties panel.

The picker should appear whenever the selection is **one logical unit** — either a single ungrouped curve or one whole chord group (which auto-expands on click via `rebuildTransformBox`). Multi-unit and partial-group selections hide the picker.

## Design

### Selection eligibility

Surface the picker iff the current selection forms a single movable unit:

- **Single ungrouped curve:** `selectedCurveIds.size === 1` and that curve has no `groupId`.
- **One complete group:** every selected curve shares the same non-null `groupId`, and that set equals the full group on the source track (no orphans, no extras).

For everything else (zero curves, multi-unit, partial group), the CURVE subsection is omitted entirely. A new helper `getMovableSelection(state): { curveIds, sourceTrack } | null` in [src/model/curve-groups.ts](src/model/curve-groups.ts) encapsulates this — checks against `state.composition.tracks[selectedTrackId].curves` so it's reusable from any future Move call sites.

### UI

In [src/ui/property-panel.ts](src/ui/property-panel.ts), when `getMovableSelection(state)` returns non-null, prepend a new `CURVE` subsection above the existing `TRACK` rows (panel header style, matching the existing `Tool Properties` / `Object Properties` headers). The subsection contains a single row:

```
Move to track  [ <select ▾> ]
```

Dropdown options, in order:
1. Every track in `comp.tracks` *except* the source track (label = track name; value = track id).
2. A separator (`<option disabled>──</option>`).
3. `+ New track` (sentinel value `__new__`).

The `<select>` defaults to a stable placeholder option (`-- Select --`, value `""`, disabled + selected). On `change`, dispatch the move and immediately reset the placeholder so the dropdown is ready for another move (the picker will re-render anyway because the selection's track changed, but resetting handles edge cases like moving back).

Use the existing `prop-row` / `prop-label` / `prop-value` CSS classes already used for the rest of the panel (see the Track/Volume rows). No new CSS.

### Move operation

Add `moveCurvesToTrack(curveIds: string[], targetTrackId: string): void` in [src/state/store.ts](src/state/store.ts). Single history snapshot, single `mutate()`:

```
history.snapshot();
store.mutate(comp => {
  const source = findTrackContaining(comp, curveIds[0]);
  const target = comp.tracks.find(t => t.id === targetTrackId);
  if (!source || !target || source === target) return;
  const moved: BezierCurve[] = [];
  for (const id of curveIds) {
    const idx = source.curves.findIndex(c => c.id === id);
    if (idx >= 0) moved.push(source.curves.splice(idx, 1)[0]!);
  }
  target.curves.push(...moved);
});
store.setSelectedTrack(targetTrackId);
// setSelectedTrack clears selectedCurveIds — re-apply so the moved curves stay selected.
store.setSelectedCurves(curveIds);
```

Notes:
- **groupId is preserved unchanged.** Per the answered design question: whole-group moves stay coupled. Single-ungrouped-curve moves had no group to begin with.
- **Curve `id`s are preserved** — this is a move, not a copy. Undo restores cleanly.
- `setSelectedTrack(targetTrackId)` updates the primary planchette's `trackId` (existing behavior in [src/state/store.ts](src/state/store.ts) `setSelectedTrack`) so subsequent perform/record uses the new track. Then re-apply the curve selection so the user can keep editing.

### "+ New track" branch

Handled at the UI layer in property-panel before delegating to `moveCurvesToTrack`. Inherits source track's tone (per answered design question) — no tone picker prompt:

```
history.snapshot();   // single snapshot covers create + move
store.mutate(comp => {
  const source = findTrackContaining(comp, curveIds[0]);
  if (!source) return;
  const newTrack = createTrack(`Track ${comp.tracks.length + 1}`, source.toneId);
  comp.tracks.push(newTrack);
  // ... same splice/push as above into newTrack
});
store.setSelectedTrack(newTrackId); // need to capture the new id outside the mutate
store.setSelectedCurves(curveIds);
```

Subtlety: `createTrack` runs inside the mutate so we need to read the new id from outside. Easiest pattern: build the track *before* the mutate, capture its id, then push it inside the mutate. That avoids a second mutate call.

To keep snapshot+mutate boundaries clean and avoid duplicating splice logic, factor a small internal helper that takes already-resolved source/target track refs and does the splice/push. Both the existing-track path and the new-track path call it.

## Edge cases

- **Source track has zero curves after move:** that's fine; empty tracks are valid.
- **Target track gets a chord group it didn't have:** also fine; groups are track-local, but moving the whole group keeps it coherent on the new track. Future operations like delete-group still work because `getGroupMembers(newTrack, anyMember)` finds them all.
- **User somehow moves a partial group via API (not UI):** the UI prevents this by hiding the picker, but `moveCurvesToTrack` itself doesn't enforce — it'd happily orphan siblings. Acceptable for now since the only call site is the UI; if needed later, add an assertion.
- **Move target deleted between dropdown open and change event:** `comp.tracks.find(...)` returns undefined; the move is a no-op. No crash, no selection change.

- **Save / load round-trip:** Move a curve, save composition, reload (File → Load) — curve is on the moved-to track in the loaded file. No schema changes, but worth confirming nothing serializes the old `selectedTrackId` accidentally.
- **Performance / record continuity:** After move, hold LMB → planchette sounds the *target* track's tone (since `setSelectedTrack` updated the primary planchette).
